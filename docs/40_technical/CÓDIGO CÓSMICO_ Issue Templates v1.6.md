## 2. Plantilla: Bug técnico

```markdown
## Descripción
<qué falla>

## Severidad
<minor/major/critical>

## Pasos para reproducir
1.
2.
3.

## Resultado actual

## Resultado esperado

## Evidencia
<logs/capturas>

## Criterio de cierre
- [ ] reproducible resuelto
- [ ] test o validación documentada
```

## 3. Plantilla: Spike técnico

```markdown
## Pregunta a resolver

## Hipótesis

## Opciones evaluadas
1.
2.
3.

## Resultado

## Recomendación

## Impacto en backlog
```

---

## 4. Campos obligatorios para P0

- prioridad,
- responsable,
- ETA,
- dependencia,
- criterio de aceptación,
- estado de riesgo.

---

## 5. Regla de calidad de issues

Una issue no entra en `ready` si carece de criterios de aceptación verificables.

---

## 6. Declaración Final

Las plantillas no son burocracia: son el contrato mínimo para reducir retrabajo y acelerar revisiones técnicas.
