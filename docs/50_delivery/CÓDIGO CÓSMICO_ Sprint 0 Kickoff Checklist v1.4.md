## 1. Propósito

Asegurar que el equipo puede iniciar Sprint 0 sin bloqueos organizativos, técnicos ni documentales.

---

## 2. Checklist de kickoff

### 2.1 Equipo y ownership

- [ ] Owners por bloque M0-M2 asignados
- [ ] Canal de comunicación operativo
- [ ] Calendario semanal de rituales confirmado

### 2.2 Entorno técnico

- [ ] Setup local reproducible validado por 2+ miembros
- [ ] Variables de entorno documentadas
- [ ] CI ejecuta lint + test smoke
- [ ] Staging accesible

### 2.3 Backlog y planificación

- [ ] Historias P0 de M0 y M1 en estado `ready`
- [ ] Dependencias críticas mapeadas
- [ ] Estimaciones revisadas por pares

### 2.4 Riesgo y calidad

- [ ] Top 5 riesgos con mitigación activa
- [ ] QA gate definido para salida de sprint
- [ ] Criterio de rollback definido para cambios críticos

### 2.5 Documentación viva

- [ ] Document Index actualizado
- [ ] Decision log al día
- [ ] Release Notes preparado para v1.4

---

## 3. Criterio de inicio de Sprint 0

Sprint 0 inicia solo si:

1. todos los ítems P0 de esta checklist están completos,
2. no hay bloqueos críticos abiertos,
3. Product y Tech Lead dan aprobación explícita.

---

## 4. Criterio de cierre de Sprint 0

- [ ] M0 completado según Technical Milestones
- [ ] M1 listo para arrancar
- [ ] Riesgo residual de arranque en nivel aceptable

---

## 5. Declaración Final

Sprint 0 no es una formalidad: es la barrera de calidad para evitar deuda temprana. Si esta checklist no se cumple, la ejecución posterior pierde predictibilidad.
