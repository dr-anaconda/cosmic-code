## 1. Uso

Este documento define gates operativos por fase para decidir PASS/FAIL sin ambiguedad.

## 2. Gate por fase

### Gate F0

- [ ] SoT por dominio documentado.
- [ ] Owners asignados para docs P0.
- [ ] Open questions con severidad activa.
- [ ] A0 abiertas = 0.

### Gate F1

- [ ] Alcance MVP incluye/no-incluye aprobado.
- [ ] KPI norte y target definidos.
- [ ] Dependencias criticas del MVP cerradas.
- [ ] No contradicciones en docs de producto y tecnica.

### Gate F2

- [ ] Art bible MVP publicada.
- [ ] Paleta, tipografia y motion rules cerradas.
- [ ] Presupuesto visual por dispositivo objetivo definido.
- [ ] Planeta + 3 biomas + VFX base especificados.

### Gate F3

- [ ] Modelo de datos MVP cerrado.
- [ ] Contratos API minimos cerrados.
- [ ] Estrategia auth (magic link) documentada.
- [ ] Eventos de analitica core definidos.

### Gate F4

- [ ] Vertical slice jugable completo.
- [ ] Flujo principal sin blockers.
- [ ] Persistencia base funcional.
- [ ] QA funcional inicial aprobada.

### Gate F5

- [ ] Tutorial contextual en 3 pasos funcional.
- [ ] Balance inicial niveles 1-4 validado.
- [ ] Instrumentacion KPI operativa.
- [ ] Bugs criticos = 0.

### Gate F6

- [ ] Build candidata de beta cerrada.
- [ ] Resultado de validacion KPI registrado.
- [ ] Decision go/no-go registrada.
- [ ] Plan de accion para Fase 2.

## 3. Regla de waiver

Solo se acepta waiver si:

- se documenta causa,
- hay accion correctiva con owner,
- existe fecha de cierre,
- Product + Tech aprueban.
